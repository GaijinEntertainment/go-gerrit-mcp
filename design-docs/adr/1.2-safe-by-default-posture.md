# Safe-by-default posture

- **Status:** accepted
- **Date:** 2026-07-10
- **Initiative:** 01-go-gerrit-mcp

Two defaults encode the safety posture. Zero configuration enables the `read` group only — write capability is always
an explicit opt-in. And the own-changes restriction (`--own-changes-only`) defaults to `true`: even with `comment` or
`transition` enabled, trail-leaving operations are refused on changes the authenticated account does not own until the
operator explicitly disables the restriction.

The second default was a genuine trade-off. Writes are already opt-in via groups, so defaulting own-changes to on
means the agent-as-reviewer use case — commenting on colleagues' changes, a primary published use case — does not work
out of the box and needs a second flag (`--own-changes-only=false`). Least-surprise argued for off; defense-in-depth
argued for on. Decided for on: an agent leaving unwanted trail on other people's changes is a reputational,
externally-visible failure, while a missing capability is a locally-discoverable inconvenience. Future readers hitting
"why doesn't my reviewer bot post?" — this is why, and it is deliberate. Flipping the default after publication would
silently degrade the safety posture of existing deployments, hence the record.
