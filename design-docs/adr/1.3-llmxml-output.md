# llmxml tool output

- **Status:** accepted
- **Date:** 2026-07-10
- **Initiative:** 01-go-gerrit-mcp

Every tool returns llmxml: an LLM-digestible subset of XML — line-structured, semantically tagged text addressed to
the model reading it. It has no transport role: nothing parses it back, there is no schema, no declaration, no
namespaces, no round-tripping — and no indentation, since models parse tag boundaries, not whitespace. Emission goes
through a purpose-built renderer, not `encoding/xml` — a serializer built for spec-compliant round-tripping is the
wrong tool for a presentation format and fights line discipline, escaping, and tag-shape choices at every step.

Considered alternatives: JSON (the ecosystem's default assumption for tool output — machine-honest but noisy for a
model to read: quoting, escaping, and depth cost attention) and markdown (reads well but delimits poorly when nesting
per-file comment threads and diff hunks). XML-style tags delimit nested structure with named boundaries, which is what
review data needs and what models parse reliably. The format is a published
behavior contract — user prompts and workflows will come to depend on its shape — so changing it later is a breaking
change, hence the record.
