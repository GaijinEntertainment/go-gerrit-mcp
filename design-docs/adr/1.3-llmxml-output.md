# llmxml tool output

- **Status:** accepted
- **Date:** 2026-07-10
- **Initiative:** 01-go-gerrit-mcp

Every tool returns llmxml: an LLM-digestible subset of XML — line-structured, semantically tagged text addressed to
the model reading it. It has no transport role: nothing parses it back, there is no schema, no declaration, no
namespaces, no round-tripping — and no indentation, since models parse tag boundaries, not whitespace. Emission goes
through a purpose-built renderer, not `encoding/xml` — a serializer built for spec-compliant round-tripping is the
wrong tool for a presentation format and fights line discipline, escaping, and tag-shape choices at every step.

Considered alternatives: JSON (the ecosystem's default assumption for tool output — machine-honest but noisy for a
model to read: quoting, escaping, and depth cost attention) and markdown (reads well but delimits poorly when nesting
per-file comment threads and diff hunks). XML-style tags delimit nested structure with named boundaries, which is what
review data needs and what models parse reliably. The format is a published
behavior contract — user prompts and workflows will come to depend on its shape — so changing it later is a breaking
change, hence the record.

**Amended 2026-07-11 — errors speak the dialect too.** Every in-band tool-call error — argument validation, client-side
refusals, Gerrit API passthrough — reaches the model as `<error tool="NAME">message</error>`, applied uniformly by
server middleware rather than per tool, so failure output never forces a parsing-mode switch. The MCP `isError` flag is
untouched (llmxml is the model-facing layer, not the transport signal), and protocol-level failures such as an unknown
tool name remain plain JSON-RPC errors. The error shape is part of the same published contract as success renders and
is pinned by golden files.

**Amended 2026-07-11 — errors carry a recovery path where one is knowable.** When the valid set behind a failed input
is known, the error names it: fuzzy `did_you_mean` proposals for file paths (large legible sets), full enumeration for
small sets (configured labels, patch-set ranges), and a `hint` field naming the tool that refreshes stale context
(reply anchors). Proposals beyond a similarity threshold are dropped — a far-fetched proposal invites a confident
wrong retry. The same stance covers misleading successes: an empty diff for a path absent from the revision becomes an
error, and scoped query results name the active `scope` so an empty page reads as "possibly out of scope", not "does
not exist". Enrichment fetches run on the failure path only.
