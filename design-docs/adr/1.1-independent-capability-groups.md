# Independent, self-sufficient capability groups

- **Status:** accepted
- **Date:** 2026-07-10
- **Initiative:** 01-go-gerrit-mcp

The server gates its tool surface through three capability groups — `read`, `comment`, `transition` — selected via
`--groups`. Groups are independent and combinable, not a cumulative ladder: each write-capable group bundles the
minimal change-read subset it needs to function alone (e.g. `comment` without `read` can still list the comments it
replies to), enabled groups union, and all three yield the full set. Tool filters (`--include-tools` /
`--exclude-tools`) apply after group resolution and only narrow — include never activates a tool outside the enabled
groups.

The considered alternative was an ordered tier ladder (`read` → `comment` → `full`), one knob with three positions —
simpler to explain, but it forces the full read surface into every write scenario and makes "comment-bot without
broad read access" inexpressible. Independent groups keep each deployment's tool count minimal (Anthropic guidance:
more tools degrade agent performance) and match the observed ecosystem gating pattern (named toolset groups).
Reversing this later would break the published config contract, hence the record.
